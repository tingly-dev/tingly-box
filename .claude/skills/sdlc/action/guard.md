# /guard

> 在约束条件下执行任务

**用途**: 有任务要执行，但想附加约束条件（保持功能一致、不破坏现有、最小变更等）。

## 使用方式

```bash
/guard [任务描述]

# 等同于说：
# "[任务描述]，但在保持功能一致的前提下"
```

## 典型场景

```bash
# CR 后修复，担心过度
/guard 修复 CR 发现的问题

# 按方案执行，附加约束
/guard 按照 design.md 实现，不改动 API

# 修复 bug，最小化变更
/guard 修复登录超时，只改必要的

# 重构，保持行为一致
/guard 重构用户模块，不改变现有行为
```

## 执行原则

使用 `/guard` 时，AI 会自动遵循：

- ✅ 只修改解决问题必需的代码
- ✅ 不改变现有功能的行为
- ✅ 不改变公共 API 接口
- ✅ 遵循现有代码风格
- ❌ 不顺便重构
- ❌ 不统一代码风格
- ❌ 不添加额外功能

## 一句话对照

| 你想说 | /guard 写法 |
|--------|-------------|
| "在保持功能一致的情况下修复" | `/guard 修复` |
| "只修必须修的，不要过度" | `/guard 修复，最小变更` |
| "按方案实现，不要偏离" | `/guard 按方案实现` |
| "修复 bug，保守点" | `/guard 修复 bug，只改必要的` |

## 示例

```bash
# CR 发现问题
/sdlc cr src/auth
# → 发现：缺少空值检查、错误未处理

/guard 修复这些问题

# AI 会：
# ✓ 只添加空值检查
# ✓ 只添加错误处理
# ✗ 不重构函数
# ✗ 不统一风格
# ✗ 不添加日志
```

## 与其他指令

| 指令 | 特点 |
|------|------|
| `/guard [task]` | 带约束执行 |
| `/sdlc coding` | 按 spec 开发 |
| `fix / refactor` | 一般执行 |
