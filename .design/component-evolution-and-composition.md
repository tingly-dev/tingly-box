# Tingly Box 组件演进与组合能力规划

> 整理日期：2026-05-21
> 定位：产品战略 / 路线图规划文档
> 配套文档：能力现状见 [`feature-highlights.md`](./feature-highlights.md)
>
> 说明：本文中「现在」标注的是代码库已实现的能力；「近期 / 中期 / 远期」为规划方向，
> 标 `（规划）` 的条目尚未实现，仅作路线参考，落地以实际排期为准。

---

## 一、整体演进主线

Tingly Box 的定位是 **Intelligence Orchestrator（智能编排器）**。所有组件围绕一条主线逐层演进：

```
接入代理  →  治理可控  →  编排自动化  →  自治协同
(Access)    (Govern)     (Orchestrate)   (Autonomy)
```

| 阶段 | 主题 | 解决的核心问题 | 主力组件 |
|------|------|--------------|---------|
| **L1 接入** | 统一接入 | "把所有模型和工具收口到一个入口" | LLM 网关、Agent 配置、OAuth 托管、多租户鉴权 |
| **L2 治理** | 可观测可控 | "让 AI 的行为可路由、可拦截、可计量" | 智能路由、护栏安全、使用分析、上下文优化 |
| **L3 编排** | 程序化驱动 | "用代码和聊天工具驱动 Agent 干活" | AgentBoot SDK、远程控制、MCP 集成 |
| **L4 自治** | 多 Agent 协同 | "多个 Agent 自主分工、互相调度" | 上述全部组合 + 编排引擎（规划） |

## 二、各组件连续性发展

每个组件按「现在 → 近期 → 中期 → 远期」描述其纵向演进，并标注它在演进中**对外释放的新接口/能力**。

### 1. LLM 网关（L1 基座）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 多协议统一代理（OpenAI / Anthropic / Gemini）、请求响应双向转换、流式、Embeddings / Images、Prompt Caching 透传 |
| **近期** | 统一 Usage 口径（跨协议归一化 token 统计）、更多原生协议（Responses API 深度支持）、请求级缓存层（规划） |
| **中期** | 语义缓存（相似请求命中缓存）、批量推理 API 聚合（规划） |
| **远期** | 网关即"模型抽象层"——上层完全不感知底层 Provider，可热插拔自研模型（规划） |
| **演进意义** | 网关越往后越成为整个系统的**协议地基**，其稳定的抽象让上层路由 / 护栏 / SDK 无需关心 Provider 差异 |

### 2. 智能路由（L2 大脑）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 规则引擎（请求类型 / 模型 / 场景条件）、成本/延迟/健康路由、Fallback、可视化 DAG、规则日志 |
| **近期** | 基于实时指标的动态权重、A/B 灰度路由（同一请求分流到不同模型做对比）（规划） |
| **中期** | 质量感知路由——结合评测分自动选模型；成本预算护栏（超预算自动降级）（规划） |
| **远期** | 自学习路由——从历史用量与结果反馈中自动优化规则（规划） |
| **演进意义** | 路由从"按规则分发"进化为"按目标优化"，是 L4 自治阶段多 Agent 调度的底层调度器 |

### 3. 护栏 & 安全（L2 闸门）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 三级决策（Allow/Review/Block）、资源访问 / 命令执行 / 内容策略、凭证保护、策略分组、远程策略注册表、变更审计 |
| **近期** | 策略与远程控制深度联动（Review 自动推 IM 审批，已具雏形）、策略命中统计看板（规划） |
| **中期** | 数据出境合规策略（PII 自动识别脱敏）、按角色的策略集（RBAC × 护栏）（规划） |
| **远期** | 策略即代码 + 策略市场，跨组织共享合规模板（规划） |
| **演进意义** | 护栏是 AI 进入生产环境的"准入闸门"，越往后越是企业级合规的核心卖点 |

### 4. 远程控制（L3 人机接口）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 9 平台 IM Bot、TOFU 配对、内联按钮审批、白名单、审计日志 |
| **近期** | 富审批卡片（展示完整 diff/命令上下文）、会话级双向对话（在 IM 里直接追问 Agent）（规划） |
| **中期** | 移动端完整 Agent 控制台（启停 / 切模型 / 看用量）、多人协同审批（规划） |
| **远期** | IM 即"指挥中心"——一个群管理一支 Agent 团队（规划） |
| **演进意义** | 远程控制是 human-in-the-loop 的承载层，让自治 Agent 在关键节点始终可被人接管 |

### 5. Agent 配置（L1 适配层）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | Claude Code / Codex / OpenCode / Xcode / VSCode / Claude Desktop 一键配置、Unified/Separate 模式、Apply/Restore |
| **近期** | 配置模板共享（团队统一下发）、配置版本与回滚（规划） |
| **中期** | 配置即 Profile——一个 Profile 打包模型+路由+护栏+MCP+Skill，整体分发（规划） |
| **远期** | 自动探测本机已装工具并智能推荐配置（规划） |
| **演进意义** | 配置层把"接入门槛"压到最低，是规模化推广（让全公司用起来）的关键 |

### 6. 使用分析（L2 度量衡）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 请求级记录、日/月聚合、缓存命中/错误率、Top 模型、按 Token 隔离、时序接口、导出 |
| **近期** | 成本归因到团队/项目/Agent、预算告警（规划） |
| **中期** | 质量度量（结合护栏命中 / 任务成功率）、ROI 看板（规划） |
| **远期** | 用量与路由/护栏形成反馈闭环，数据反哺自动优化（规划） |
| **演进意义** | 分析是所有"自动优化"的数据源，没有度量就没有 L4 的自学习 |

### 7. MCP 集成（L3 工具生态）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 内置 web_search/web_fetch、自定义 MCP Client、工具可见性控制、目录安装 |
| **近期** | MCP 工具市场（一键安装企业常用工具）、工具调用纳入护栏与审计（规划） |
| **中期** | MCP 工具的用量计量与成本分摊、工具级权限（哪个 Agent 能用哪个工具）（规划） |
| **远期** | MCP 成为"Agent 的能力插槽"，工具可被路由/编排动态注入（规划） |
| **演进意义** | MCP 决定 Agent "能做多少事"，是扩展能力边界的标准接口 |

### 8. 上下文优化（L2 效率层）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | Smart Compact（thinking 压缩、对话去重、文档/错误精简、XML 压缩） |
| **近期** | 自适应压缩强度（按窗口剩余动态调节）、压缩前后可视化对比（规划） |
| **中期** | 跨会话记忆层（把历史结论沉淀为可检索记忆而非全量保留）（规划） |
| **远期** | 与 MCP 记忆工具结合，形成长期记忆体系（规划） |
| **演进意义** | 上下文优化决定长任务的"耐力"，是 Agent 能否跑长流程的关键 |

### 9. 多租户 & 鉴权（L1 安全基座）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 双 Token 隔离、随机生成、JWT、Token 轮转、多 API Token 独立计量 |
| **近期** | 基于角色的访问控制（RBAC）、Token 级配额限制（规划） |
| **中期** | 组织/团队/项目层级、SSO 对接（规划） |
| **远期** | 完整的企业身份体系（SCIM/审计合规）（规划） |
| **演进意义** | 鉴权是 SaaS 化与企业落地的底座，决定能否多组织安全共用一套部署 |

### 10. OAuth 托管（L1 凭证层）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | Claude Code / Codex / Kimi 一键授权、自动续期、状态可视化、代理支持 |
| **近期** | Gemini CLI / Qwen Code OAuth（规划）、授权额度可见（剩余 quota）（规划） |
| **中期** | 团队级共享授权池 + 用量分摊、多账号轮换避免限流（规划） |
| **远期** | 订阅额度与 API Key 混合调度（按成本最优自动选）（规划） |
| **演进意义** | OAuth 托管让"订阅制额度"可被编排，是降本的关键杠杆 |

### 11. AgentBoot SDK（L3 编排引擎）

| 阶段 | 能力演进 |
|------|---------|
| **现在** | 统一 Execute、Stream 事件流、权限/Ask 回调、会话恢复、MCP/API/Skill 完整透传、Driver+Transport 可扩展 |
| **近期** | Codex / Gemini / Cursor Driver 落地、多 Agent 并发编排原语（规划） |
| **中期** | Agent 工作流引擎（DAG 编排多个 Agent 步骤）、Agent 间消息传递（规划） |
| **远期** | 自治 Agent 编排——Agent 可动态拉起子 Agent、自主分工（规划） |
| **演进意义** | SDK 是从"工具"走向"平台"的引擎，L4 自治阶段的所有编排都建立在它之上 |

## 三、组合产生的能力（组件 × 组件）

单个组件解决单点问题，**真正的护城河来自组合**。下表列出已可落地或近期可达的关键组合。

| 组合 | 涌现能力 | 状态 |
|------|---------|------|
| 网关 + 智能路由 | **成本/质量自适应推理**：同一接口背后按请求特征自动选最优模型 | 现在 |
| 网关 + 多租户 + 使用分析 | **可计量的内部 AI 平台**：多团队共用一套部署，用量与成本按 Token 精确分摊 | 现在 |
| 护栏 + 远程控制 | **带人工审批的安全执行**：高危操作自动 Review 并推 IM 审批，人点一下放行 | 现在（雏形） |
| 护栏 + 凭证保护 + 网关 | **合规可信代理**：敏感数据脱敏出境、危险操作拦截，满足金融/医疗合规 | 现在 |
| SDK + MCP + Skill | **可嵌入的领域 Agent**：把 Claude Code 装进自己的系统，并注入私有知识库与规范 | 现在 |
| SDK + 护栏 + 远程控制 | **无人值守但可接管的自动化**：Agent 自动跑流程，关键节点回调人工审批 | 现在 |
| SDK + 会话恢复 + 上下文优化 | **可断点续跑的长任务**：超时/中断后从断点续跑，长流程不丢上下文 | 现在 |
| OAuth 托管 + 智能路由 | **订阅额度智能调度**：在订阅额度与 API 计费间按成本最优分流 | 中期（规划） |
| 路由 + 分析 + 护栏（反馈闭环） | **自优化编排**：用量与结果反哺路由策略，系统越用越聪明 | 远期（规划） |
| SDK + 路由 + 远程控制（全栈） | **Agent 团队指挥中心**：一个 IM 群调度一支自主分工的 Agent 队伍 | 远期（规划） |

### 组合能力的"叠加效应"图谱

```
                    ┌─────────────────────────────────────┐
   L4 自治          │   Agent 团队指挥中心 / 自优化编排      │  ← 全栈组合
                    └─────────────────────────────────────┘
                                    ▲
   L3 编排    ┌──────────┐   ┌──────────┐   ┌──────────┐
              │ SDK+MCP  │   │ SDK+护栏 │   │ SDK+会话 │     ← 程序化驱动
              │ 领域Agent│   │ +远程审批│   │ 续跑长任务│
              └──────────┘   └──────────┘   └──────────┘
                                    ▲
   L2 治理    ┌──────────┐   ┌──────────┐   ┌──────────┐
              │网关+路由 │   │护栏+凭证 │   │网关+租户 │     ← 可控可计量
              │自适应推理│   │合规代理  │   │+分析平台 │
              └──────────┘   └──────────┘   └──────────┘
                                    ▲
   L1 接入    ┌─────────────────────────────────────────┐
              │  网关 / Agent配置 / OAuth托管 / 多租户鉴权  │   ← 统一接入基座
              └─────────────────────────────────────────┘
```

## 四、适配场景（按客户类型 × 组合）

| 目标客户 | 核心诉求 | 推荐组合 | 落地阶段 |
|---------|---------|---------|---------|
| **个人开发者 / 小团队** | 省钱、多工具统一、订阅额度复用 | 网关 + Agent 配置 + OAuth 托管 | 现在 |
| **AI 应用创业公司** | 模型可热切换、成本可控、快速迭代 | 网关 + 智能路由 + 使用分析 | 现在 |
| **中大型研发团队** | 全员统一接入、用量分摊、配置标准化 | 网关 + 多租户 + Agent 配置 + 分析 | 现在 |
| **金融 / 医疗等合规行业** | 数据不出境、操作可审计、人工兜底 | 护栏 + 凭证保护 + 远程控制 + 审计 | 现在 |
| **平台 / 自动化团队** | 把 Agent 嵌进自有系统做自动化 | AgentBoot SDK + MCP + 护栏回调 | 现在 |
| **DevOps / SRE** | 无人值守自动化 + 移动端可接管 | SDK + 远程控制 + 护栏 Review | 现在（雏形） |
| **大型企业 IT** | 多组织、SSO、RBAC、合规 | 多租户 + 鉴权（RBAC/SSO）+ 护栏 | 中期（规划） |
| **AI 中台 / 平台型公司** | 自主 Agent 编排、自优化 | 全栈组合 + 编排引擎 | 远期（规划） |

## 五、规划落地优先级建议

按「价值 × 可达性」排序，建议近期优先推进以下三条线，它们都建立在已有组件上、能快速放大组合价值：

1. **护栏 ↔ 远程控制闭环**（L2×L3）：把 Review 决策完整接到 IM 富审批卡片——这是"安全可信 AI"最直观的卖点，且雏形已具。
2. **SDK 多 Agent 编排原语**（L3）：在现有 Execute/会话能力上加并发编排，打开"平台化"想象空间。
3. **使用分析 → 成本归因 + 预算告警**（L2）：把已有的用量数据变成可分摊、可预警的成本治理能力，直击企业付费动机。
