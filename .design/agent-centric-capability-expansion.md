# Tingly Box 能力展开：从 Agent 三件套向外生长

> 整理日期：2026-05-21
> 定位：以「驱动 Agent」为核心叙事，从三个起点能力逐层展开整个平台
> 配套文档：能力清单见 [`feature-highlights.md`](./feature-highlights.md)；演进路线见 [`component-evolution-and-composition.md`](./component-evolution-and-composition.md)

---

## 0. 一句话主张

> **Tingly Box 的本质是「驱动 Agent」。**

一切能力都围绕一个中心命题展开：**怎样把 AI Agent 用起来、用得稳、用得起规模**。
我们提供三种互补的「驱动入口」——它们是本平台的起点与重点，其余能力都是为放大这三者而存在的支撑环。

| 驱动入口 | 谁来驱动 | 形态 | 一句话 |
|---------|---------|------|--------|
| **AgentBoot SDK** | 程序 / 系统 | Go 代码 | 把 Agent 嵌进你的系统，自动化地驱动 |
| **远程控制（Remote）** | 人 | IM 聊天 | 人在手机/群里随时驱动与接管 Agent |
| **Agent 适配** | 现成 AI 工具 | 一键配置 | 让你已经在用的 Claude Code / Codex 等接入并被驱动 |

> 三个入口，**同一个 Agent 执行核心**。差别只在"由谁发起、以什么形态"。

## 1. 起点：Agent 三件套（核心）

```
              ┌──────────────────────────────────────┐
              │            Agent 执行核心              │
              │   （启动 / 事件流 / 权限 / 会话）       │
              └──────────────────────────────────────┘
                  ▲             ▲              ▲
        ┌─────────┘             │              └─────────┐
   ┌──────────┐          ┌──────────┐            ┌──────────┐
   │ AgentBoot│          │  Remote  │            │ Agent 适配│
   │   SDK    │          │  控制    │            │  (配置)   │
   │ 程序驱动 │          │  人驱动  │            │ 工具接入  │
   └──────────┘          └──────────┘            └──────────┘
   开发者/平台            运维/管理者              工程师/终端
```

### 1.1 AgentBoot SDK —— 程序化驱动
- **是什么**：Go SDK，统一 `Execute` 入口启动 Claude Code 等 Agent，返回有序事件流，内联处理权限/Ask 回调，支持会话恢复。
- **关键点**：**完整透传** Agent 原生能力（MCP / 模型 API / Skill），不是裁剪子集。
- **解决**：把 Agent 变成可被任意系统调用的"函数"——工单系统、CI、内部平台都能程序化地驱动它。

### 1.2 远程控制（Remote）—— 人驱动 / 人接管
- **是什么**：6 平台 IM Bot（飞书 / 钉钉 / 企业微信 / 微信 / Telegram / Lark），TOFU 配对、内联按钮审批、白名单、审计。
- **关键点**：把"人"放进 Agent 的执行回路——尤其是**关键节点的人工审批与随时接管**。
- **解决**：Agent 自动跑，但人始终能在手机上看到、批准、叫停，不必守在终端前。

### 1.3 Agent 适配 —— 让现成工具接入
- **是什么**：为 Claude Code / Codex / OpenCode / Xcode / VSCode / Claude Desktop 提供一键配置，Apply/Restore。
- **关键点**：**零迁移成本**复用用户已经在用的工具，让它们的流量进入 Tingly Box 被统一驱动与治理。
- **解决**：不要求用户改用新工具，而是让旧工具"插上"平台能力。

> **三件套的共性**：它们都是「把 Agent 跑起来」的入口。一旦 Agent 跑起来，下面的支撑环就开始发挥价值。

## 2. 第一层展开（赋能环）：让 Agent "能干活"

Agent 被驱动起来后，立刻需要三样东西才能真正干活——**模型、工具、耐力**。

```
   [ Agent 三件套 ]
         │  Agent 需要……
         ├──────────────┬───────────────┐
     给它模型        给它工具         给它耐力
   ┌──────────┐   ┌──────────┐   ┌────────────┐
   │ LLM 网关 │   │ MCP 网关 │   │ 上下文优化 │
   └──────────┘   └──────────┘   └────────────┘
```

| 支撑能力 | 为三件套补足什么 | 与核心的连接 |
|---------|----------------|-------------|
| **LLM 网关** | 模型供给：一个端点接管所有 Provider，协议自动转换 | SDK / 适配 / Remote 驱动的 Agent，请求都从网关出去 |
| **MCP 网关** | 工具供给：聚合 web 搜索、私有知识库、外部服务等多个 MCP Server 到一个入口 | SDK 可把 MCP **完整透传**给 Agent；网关统一发现/执行/管控工具 |
| **上下文优化** | 续航能力：Smart Compact 压缩去重，长任务不中断 | 让 SDK 跑的长流程、适配工具的长会话都能撑下去 |

> **此环结论**：有了模型+工具+耐力，三件套驱动的 Agent 从"能启动"升级为"能完成真实任务"。

---

## 3. 第二层展开（管控环）：让 Agent "干得对、可信"

能干活之后，下一个问题是**别干坏事、别乱花钱**。管控环让 Agent 的行为可约束、可优化。

```
   [ 三件套 + 赋能环 ]
         │  Agent 的行为需要……
         ├────────────────────┐
     约束边界              选对模型
   ┌──────────┐         ┌──────────┐
   │ 护栏安全 │◄───────►│ 智能路由 │
   └────┬─────┘         └──────────┘
        │ Review 决策
        ▼
   ┌──────────┐
   │  Remote  │  ← 回到核心：人工审批闭环
   └──────────┘
```

| 支撑能力 | 为三件套补足什么 | 与核心的连接 |
|---------|----------------|-------------|
| **护栏 & 安全** | 行为边界：危险操作 Block / 凭证脱敏 / 高危 Review | **护栏的 Review ↔ Remote 形成闭环**：触发审批 → 推 IM → 人点批准 |
| **智能路由** | 决策优化：按请求类型/成本/健康自动选模型、Fallback | 三件套的每次模型调用都经路由择优，主 Agent 用强模型、subagent 用便宜模型 |

> **此环亮点：护栏 × Remote 的闭环**——这正是三件套互相强化的典型例子：
> SDK/适配驱动 Agent 自动跑 → 撞到护栏的 Review → 通过 Remote 推给人 → 人在 IM 一键放行 → Agent 继续。
> "自动化"与"人可控"在这里合二为一。

---

## 4. 第三层展开（运营环）：让 Agent "可规模化运营"

当 Agent 在团队/公司层面铺开，需要解决**多人多团队的隔离、计费、凭证与成本治理**。

```
   [ 三件套 + 赋能环 + 管控环 ]
         │  规模化需要……
         ├──────────────┬───────────────┐
     身份与隔离      凭证供给         成本与度量
   ┌────────────┐  ┌──────────┐   ┌──────────┐
   │ 多租户鉴权 │  │ OAuth托管│   │ 使用分析 │
   └────────────┘  └──────────┘   └──────────┘
```

| 支撑能力 | 为三件套补足什么 | 与核心的连接 |
|---------|----------------|-------------|
| **多租户 & 鉴权** | 隔离：双 Token 分离，多 Token 独立配额与计量 | 不同团队/Agent 用不同 Token 驱动，互不影响、各自计费 |
| **OAuth 托管** | 凭证：托管订阅额度授权并自动续期 | 三件套驱动的 Agent 共享一份订阅授权，降本且不断线 |
| **使用分析** | 度量：Token/费用/延迟全记录，按 Token 分摊 | 无论 SDK / Remote / 适配 哪种入口产生的调用，都进同一本账 |

> **此环结论**：三件套从"团队能用"升级为"公司能管"——可隔离、可计费、可审计。

---

## 5. 全景：同心圆能力地图

```
                    ┌─────────────────────────────┐
                    │   运营环：多租户 / OAuth / 分析 │   可规模化运营
                  ┌─┴─────────────────────────────┴─┐
                  │   管控环：护栏安全 / 智能路由       │   干得对、可信
                ┌─┴─────────────────────────────────┴─┐
                │  赋能环：LLM网关 / MCP网关 / 上下文优化 │   能干活
              ┌─┴─────────────────────────────────────┴─┐
              │        ★ 核心：Agent 三件套 ★            │   把 Agent 驱动起来
              │   SDK（程序） · Remote（人） · 适配（工具） │
              └─────────────────────────────────────────┘
```

**读法**：从中心向外，是 Agent 从「跑起来 → 能干活 → 干得对 → 可运营」的成熟度递进；
每一环都不是独立功能，而是**为放大核心三件套而存在的支撑**。

## 6. 渐进式落地路径（建议按展开顺序采纳）

这套展开顺序同时也是推荐的**采纳路径**——从核心入口切入，按需向外加环。

| 阶段 | 采纳能力 | 达成状态 | 适合谁 |
|------|---------|---------|--------|
| **第 1 步：选一个入口** | 三件套任选其一（如先用 Agent 适配让团队接入 Claude Code） | Agent 跑起来了 | 任何团队 |
| **第 2 步：补赋能环** | + LLM 网关（统一模型）+ MCP（私有工具）+ 上下文优化 | Agent 能完成真实任务 | 开始认真用 AI 的团队 |
| **第 3 步：补管控环** | + 护栏（边界）+ 智能路由（择优），并把护栏 Review 接到 Remote | Agent 可信、可控、省钱 | 进生产/合规环境 |
| **第 4 步：补运营环** | + 多租户（隔离计费）+ OAuth 托管（凭证）+ 使用分析（成本治理） | Agent 在公司级规模化运营 | 全员推广 / 平台化 |
| **第 5 步：组合放大** | 三件套全开 + 全环联动（如 SDK 编排 + Remote 审批 + 路由择优 + 分析归因） | Agent 团队级自动化 | AI 中台 / 平台型公司 |

## 7. 三件套互相强化的典型组合

核心入口之间、入口与支撑环之间的组合，才是真正的差异化。

| 组合 | 涌现能力 | 状态 |
|------|---------|------|
| **SDK + Remote** | 无人值守但可接管：程序驱动 Agent 自动跑，关键节点回调人工审批 | 现在 |
| **适配 + Remote** | 现成工具也能被远程管控：Claude Code 的高危操作经护栏推到 IM 审批 | 现在（雏形） |
| **SDK + 适配** | 同核异形：本地工程师用适配跑 Claude Code，平台用 SDK 跑同一个核心，行为一致 | 现在 |
| **SDK + MCP + Skill** | 可嵌入的领域 Agent：把 Claude Code 装进自有系统并注入私有工具与规范 | 现在 |
| **SDK + 会话恢复 + 上下文优化** | 可断点续跑的长任务：超时中断后从断点续跑，长流程不丢上下文 | 现在 |
| **三件套全开 + 全环** | Agent 团队指挥中心：一个 IM 群调度一支被 SDK 编排、被路由择优、被分析归因的 Agent 队伍 | 远期（规划） |
